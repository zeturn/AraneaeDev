title: Go Fiber 重构实施计划
---

本文档用于落地 Araneae 后端 Go Fiber 重构，目标是在不影响现有 Django + Flask 运行的前提下，逐步完成能力迁移。

## 目标与约束

- 保持前端视觉风格一致（无边框风格不变）
- 支持定时任务、手动触发任务、API 触发任务
- 支持用户提前上传项目代码并绑定任务执行版本
- 提供完整用户权限控制（认证、授权、资源隔离、审计）
- 建立完整日志体系（访问、业务、任务执行、审计）
- 采用并行迁移：新 Go Fiber 服务与现有 ControlNode 并行运行，逐步切流

## 分阶段计划

### Phase 0：现状梳理与冻结（1 周）

- 梳理现有接口、任务流、权限点、日志点
- 形成“旧接口 -> 新接口”的映射清单
- 明确迁移边界：优先迁移只读/低风险 API，再迁移任务写操作
- 输出回滚预案与灰度切流策略

交付物：
- 接口映射表
- 数据模型映射表
- 风险清单与回滚手册

### Phase 1：Go Fiber 基础骨架（1 周）

- 新建独立 Go Fiber 服务（独立端口，不影响现网）
- 完成配置加载、路由分组、中间件、健康检查
- 接入统一日志（JSON 结构化日志，包含 request_id）
- 增加基础可观测性埋点（状态、耗时、错误）

交付物：
- 可运行的 Go Fiber 服务基础框架
- 健康检查与基础中间件

### Phase 2：认证与权限（1~2 周）

- 接入 JWT 鉴权（含刷新策略）
- 落地 RBAC：用户-角色-权限-资源四层模型
- 增加资源级隔离：仅允许访问本人/本团队资源
- 关键操作记录审计日志（谁在何时操作了什么资源）

交付物：
- 统一认证中间件
- 权限检查中间件
- 审计日志模型与查询接口

### Phase 3：任务系统迁移（2 周）

- 设计统一触发入口 `TaskDispatcher`
- 实现三种触发方式：
  - 定时触发（cron）
  - 手动触发（UI 按钮）
  - API 触发（外部调用，支持幂等键）
- 增加分布式锁与并发策略（串行/并行/跳过重复）：
  - 采用 Redis 锁（SET NX EX）作为主方案， 锁键粒度建议为 `task:{task_id}:dispatch`
  - 通过短 TTL + 执行完成主动释放，避免死锁
  - 对锁竞争失败请求返回可观测状态码与提示，便于上层重试或降级
- 完成任务执行状态机：pending/running/success/failed/timeout

交付物：
- 任务触发与调度核心模块
- 任务执行记录与重试机制

### Phase 4：项目代码上传与执行（2 周）

- 提供代码上传接口（zip/tar.gz）和版本管理
- 任务执行绑定具体版本（或 latest 策略）
- 执行端拉取对应版本代码并执行，回传 stdout/stderr
- 增加执行安全策略：
  - 上传阶段校验： 文件类型白名单（zip/tar.gz）、大小上限、解压后总文件数与总大小上限（防 zip bomb）
  - 上传内容安全扫描： 基础恶意文件特征扫描（可对接杀毒引擎或企业安全网关）
  - 资源限制 （CPU/内存/超时）
  - 命令白名单/模板化
  - 默认隔离执行环境（建议容器化）

交付物：
- 代码版本仓储与元数据管理
- 基于版本的任务执行链路

### Phase 5：前端兼容与灰度切流（1~2 周）

- 保持现有前端无边框风格，避免 UI 结构性变更
- 先做后端接口兼容层，减少前端改动量
- 按模块灰度切流，逐步从旧服务迁移到 Go Fiber
- 监控错误率、耗时、任务成功率，满足阈值后全量切换

交付物：
- 兼容层 API
- 灰度切流配置与观测看板

## 功能验收标准

### 1) 前端风格一致（无边框）

- 页面主题变量与布局不变
- 核心页面截图比对通过
- 用户交互路径与旧版一致

### 2) 三种任务触发全部可用

- 定时任务可创建、启停、修改 cron
- 手动触发可即时执行
- API 触发具备鉴权与幂等保障

### 3) 代码上传并绑定任务执行

- 用户可上传项目代码并生成版本号
- 任务可指定版本运行
- 执行日志可按任务实例查询

### 4) 完整权限控制

- 未授权访问全部拒绝
- 越权访问资源被拦截
- 关键操作具备审计追踪

### 5) 日志完整

- 访问日志、业务日志、审计日志、执行日志齐全
- 可按 request_id / task_run_id 关联追踪
- 敏感字段已脱敏

## 风险与应对

- 风险：双栈并行期间接口行为不一致
  - 应对：建立契约测试与回归清单
- 风险：任务重复触发或并发冲突
  - 应对：分布式锁 + 幂等键 + 状态机约束
- 风险：用户代码执行安全
  - 应对：隔离执行环境 + 资源上限 + 运行时审计

## 建议优先级

1. 先完成 Go Fiber 基础骨架与日志
2. 再迁移认证与权限
3. 然后迁移任务触发与调度
4. 最后迁移代码上传执行链路与灰度切流

以上顺序可确保每一步都可回滚、可观测、可验证。
